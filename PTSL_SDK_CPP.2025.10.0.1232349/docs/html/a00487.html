Changelog

Command Progress Tracking

There are a few behavioral changes associated with the command progress tracking updates.

  1. Status Messages. For some commands you might notice a series of status messages being held by the server until command completion. These messages may appear as info items in the error list of the final command response but will not be flagged as an error or have any other impact on the response. Once more progress tracking related changes are added, these status updates will be returned to you as they are logged by the host.
  2. Host Busy State. If the host receives commands while certain modal dialogs or progress bars are being displayed by the host GUI, an info message will be logged indicating that the host is busy and that the command will wait for the current action to conclude before starting. These commands will go into a "queued" state.
  3. Command Timeout. Commands can now timeout while in progress and will receive a new PT_CommandTimeout command error upon termination. Legacy clients will receive a PT_UnknownError instead of the new error type, but the accompanying error message will still indicate what happened. Note, it's possible for any "queued" command to timeout while waiting to run. When this happens, in addition to the PT_CommandTimeout command error, there will be a PT_HostIsBusy command error in the error list of the response so that it can be distinguished from the general timeout case. The default timeout value for the current release is 24 hours.

Please note, that the new command progress tracking infrastructure is implemented on the host side which means that it will be automatically applied to all clients.

C++ Client Wrapper Changes

  1. Deprecation of Individual Command Structures. To simplify maintenance, all individual command structures and handlers are deprecated. The deprecated functionality will remain for existing clients, but will not support future API changes.
  2. New SendRequest Method. A newly introduced SendRequest method takes a CppPTSLRequest object, which in turn requires a CommandId and a JSON string representing the command request body. This allows new commands to be submitted without a command-specific handler. The example request and response JSON strings are provided in the documentation for each command. This method is safe to use with all existing commands and is fully backward compatible.
  3. OnResponseReceived has been updated to use SendRequest based structures. If you want to use the old version of the callback you can use the OnResponseReceived_V1_Deprecated callback. Note, that the old callback can only be used with the old command methods while the new callback can only be used with the new command methods (i.e. SendRequest).
  4. The documentation is moved from the C++ client to the proto file. See CommandId for more details and examples.
  5. The client now depends on the nlohmann_json library version 3.11.3. The library is not exposed in public headers.

These changes also affect ptslcmd example application as it's now using the new SendRequest method instead of individual command structures. If you need to build an old version of the ptslcmd for this release, you can use ptslcmd.sdk.2024.06.0 target in the build script.
