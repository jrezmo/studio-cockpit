Pro Tools 2025.\+06 is compatible with clients using \mbox{\hyperlink{a02036_a1403c9ca4289d8a9417e8c4da2b94c0e}{Major Version}} 1 and higher.

For further information, refer to the Versioning Changes sections.\hypertarget{a00488_release_notes_2025_06_changelog}{}\doxysubsection{Changelog}\label{a00488_release_notes_2025_06_changelog}
\hypertarget{a00488_release_notes_2025_06_changelog_versioning_changes}{}\doxysubsubsection{Versioning Changes}\label{a00488_release_notes_2025_06_changelog_versioning_changes}
\hypertarget{a00488_release_notes_2025_06_changelog_versioning_changes_version_triplet}{}\doxyparagraph{Version Triplet}\label{a00488_release_notes_2025_06_changelog_versioning_changes_version_triplet}
Starting with the 2025.\+06 release, the PTSL \mbox{\hyperlink{a02036_a1403c9ca4289d8a9417e8c4da2b94c0e}{version}} has been updated from 5 (2024.\+10) to 2025 (2025.\+XX). Additionally, \mbox{\hyperlink{a02036_a09c035a9bbb8296a09dfcfcc0c62de56}{version\+\_\+minor}} (6) and \mbox{\hyperlink{a02036_ac3eea2a8ad35f84eeb9f00174f197679}{version\+\_\+revision}} (0) are now used to differentiate releases and subreleases within the year.

Old clients (prior to this release) will receive only the major \mbox{\hyperlink{a02048_a1403c9ca4289d8a9417e8c4da2b94c0e}{version}} (2025) for all Pro Tools releases in 2025.

New clients (starting from this release) can process the entire triplet (\mbox{\hyperlink{a02048_a1403c9ca4289d8a9417e8c4da2b94c0e}{version}}, \mbox{\hyperlink{a02048_a09c035a9bbb8296a09dfcfcc0c62de56}{version\+\_\+minor}}, \mbox{\hyperlink{a02048_ac3eea2a8ad35f84eeb9f00174f197679}{version\+\_\+revision}}) to differentiate between Pro Tools releases. Older Pro Tools releases only have the major version, i.\+e., 5.\+0.\+0 for 2024.\+10.\hypertarget{a00488_release_notes_2025_06_changelog_versioning_changes_protocol}{}\doxyparagraph{Protocol}\label{a00488_release_notes_2025_06_changelog_versioning_changes_protocol}
Starting with the 2025.\+06 release, the SDK package contains multiple {\ttfamily .proto} files. Each file includes the full protocol description for a release. For example, PTSL.\+2024.\+10.\+0.\+proto contains the protocol definition for the 2024.\+10 release, wrapped inside {\ttfamily package ptsl\+\_\+2024\+\_\+10\+\_\+0}. The current release is always named {\ttfamily \mbox{\hyperlink{a00464}{PTSL.\+proto}}}, wrapped inside {\ttfamily package ptsl}.

Using the \mbox{\hyperlink{a01328_a2c6a9bfa639cab78fcb3d4159c460c5f}{Send\+Request}} and \mbox{\hyperlink{a01980}{Cpp\+PTSLRequest}}, it is possible to send requests from different releases to the same Host. The Host response is adapted to the capabilities and protocol of the release requested by the client.

There are two known limitations to this\+:
\begin{DoxyItemize}
\item Host-\/side versioning is an area of active work\+:
\begin{DoxyItemize}
\item Some deprecated enumeration values may be prioritized over their new replacements regardless of the request version.
\item Some commands may return both old properties and their new replacements.
\end{DoxyItemize}
\item Host-\/side versioning is a relatively new addition\+:
\begin{DoxyItemize}
\item It is not always possible to request API behavior similar to an older release, e.\+g., 2023.\+03.
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{a00488_release_notes_2025_06_changelog_versioning_changes_cpp_client}{}\doxyparagraph{C++ Client}\label{a00488_release_notes_2025_06_changelog_versioning_changes_cpp_client}
The C++ client reflects the changes mentioned above. Both \mbox{\hyperlink{a01980}{Cpp\+PTSLRequest}} and \mbox{\hyperlink{a01984}{Cpp\+PTSLResponse}} provide the ability to get and set the version triplet.

Old command-\/specific methods are now pinned to the 2024.\+10 PTSL version and are no longer expected to be updated.\hypertarget{a00488_release_notes_2025_06_changelog_enumerations_changes}{}\doxysubsubsection{Enumeration Changes}\label{a00488_release_notes_2025_06_changelog_enumerations_changes}
There are several changes to enumerations to reduce name clashes and make their behavior more uniform\+:
\begin{DoxyItemize}
\item All enumeration values now have a prefix based on their type name.
\begin{DoxyItemize}
\item The old values are deprecated but still accepted and can be returned by the Host.
\end{DoxyItemize}
\item All enumeration types (except \mbox{\hyperlink{a00494_aaf8d2069e2c6bbecf4aa8f9819028e8a}{Command\+Id}}) now start with the \textquotesingle{}$\ast$\+\_\+\+Unknown value = 0\textquotesingle{}.
\begin{DoxyItemize}
\item This may offset values in some types, but it will not affect communication with the Host as only the string names are used.
\end{DoxyItemize}

\end{DoxyItemize}\hypertarget{a00488_release_notes_2025_06_changelog_intermediate_responses}{}\doxysubsubsection{Intermediate Responses}\label{a00488_release_notes_2025_06_changelog_intermediate_responses}
This release enhances support for intermediate responses during command execution. A single task can now emit multiple {\ttfamily In\+Progress} intermediate responses, providing improved progress tracking and feedback. These improvements are applied universally to all clients from the Pro Tools side, enabling even older SDK versions to process them if desired. While intermediate response capability existed in the very first SDK release, the 2025.\+06 update delivers Pro Tools-\/side enhancements that improve functionality across all client implementations. New clients built with the updated SDK can now process intermediate responses seamlessly via updated \mbox{\hyperlink{a01328_a2c6a9bfa639cab78fcb3d4159c460c5f}{Send\+Request}}.

Key updates\+:


\begin{DoxyItemize}
\item Intermediate responses may include a JSON body, typically reserved for specific APIs such as the Event System (see below).
\item The final response consolidates all accumulated data, including errors, warnings, and informational messages generated during execution.
\end{DoxyItemize}

C++ Client\+:
\begin{DoxyItemize}
\item The \mbox{\hyperlink{a01328_a2c6a9bfa639cab78fcb3d4159c460c5f}{Send\+Request}} method has been updated to include a new callback mechanism. This enables clients to receive all responses associated with a task, including intermediate responses, in real-\/time.
\end{DoxyItemize}\hypertarget{a00488_release_notes_2025_06_changelog_event_system}{}\doxysubsubsection{Event System (\+Experimental)}\label{a00488_release_notes_2025_06_changelog_event_system}
The 2025.\+06 release introduces the Event System, enabling Pro Tools to send real-\/time notifications to subscribed clients. This system allows clients to subscribe to specific events, unsubscribe from them, and receive event notifications as they occur.\hypertarget{a00488_autotoc_md17}{}\doxyparagraph{Workflow}\label{a00488_autotoc_md17}

\begin{DoxyEnumerate}
\item \mbox{\hyperlink{a00494_aaf8d2069e2c6bbecf4aa8f9819028e8aa3123650ddf7f82ade0c8def87b1f7f40}{Subscribe\+To\+Events}}\+:
\begin{DoxyItemize}
\item Adds specified events to the subscription list for the current PTSL session.
\item Each subscription is uniquely identified by the combination of PTSL session, Event\+Id, and filter data. Any difference in filter data results in a new subscription, which is managed independently, even if they share the same Event\+Id.
\item For example, a hypothetical event {\ttfamily EId\+\_\+\+Track\+Record\+Enabled\+State\+Changed} could include a track ID in the filter data to listen for changes to the record-\/enabled state of a specific track.
\end{DoxyItemize}
\item \mbox{\hyperlink{a00494_aaf8d2069e2c6bbecf4aa8f9819028e8aa4890bca2c334308b21af02d39e25b1ee}{Unsubscribe\+From\+Events}}\+:
\begin{DoxyItemize}
\item Removes specified events (and their associated filter data) from the subscription list for the current PTSL session.
\end{DoxyItemize}
\item \mbox{\hyperlink{a00494_aaf8d2069e2c6bbecf4aa8f9819028e8aabc3d3419d76198a03985575dff948bce}{Poll\+Events}}\+:
\begin{DoxyItemize}
\item A long-\/polling API that streams real-\/time events to the client.
\item Operates indefinitely without a typical end condition.
\item Sends events as they are reported by Pro Tools, using the Intermediate Responses mechanism.
\item Only one active Poll\+Events connection is allowed per PTSL session. If a new Poll\+Events call is made, the previous connection is terminated, and polling is transferred to the new connection.
\item All event-\/related APIs, including Poll\+Events, are processed with higher execution priority compared to normal API commands, ensuring timely delivery of events without blocking other API commands.
\end{DoxyItemize}
\end{DoxyEnumerate}\hypertarget{a00488_autotoc_md18}{}\doxyparagraph{Supported Events in 2025.\+06}\label{a00488_autotoc_md18}
